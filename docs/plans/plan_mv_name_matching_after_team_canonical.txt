0) Goal
- Improve name-matching quality in profile MVs after `team_canonical` is available.
- Reduce false positive player/team matches from name-only fallback.
- Keep current MVs backward compatible during rollout.

1) Current State (as-is)
- `mv_player_profiles` uses 2 layers:
  - Layer 1: exact via `player_crossref.tm_player_id`.
  - Layer 2: fallback by normalized `player_name` only.
- `mv_team_profiles` fallback is normalized team name join from `standings` -> `team_metadata`.
- `team_canonical` now exists with `fbref_team_id`, `tm_team_id`, and aliases.
- Gap: fallback matching does not enforce canonical team context.

2) Problem Statement
- Name-only player fallback can attach wrong Transfermarkt row when same/similar names exist.
- Team name normalization alone can still mismatch abbreviations or locale variants.
- Current MVs do not expose match confidence/quality signals.

3) Design Decision Summary
- Keep strict ID match as highest priority.
- Add team-constrained fallback using `team_canonical` before any loose name fallback.
- Introduce explicit match metadata (`matched_by`, `match_confidence`) in MV layer.
- Roll out in shadow mode (`_v2` MVs), compare with existing, then swap.

4) Proposed Data Model & Logic
4.1 New helper MV (optional but recommended)
- `mv_tm_player_candidates` precomputes normalized candidate rows from `market_values`:
  - `tm_player_id`, `league_id`, `tm_team_id`, `player_name_norm`, `team_name_norm`,
    `market_value_numeric`, `player_image_url`, `loaded_at`.
- Dependency rule (critical): if this helper MV is enabled, all downstream refreshes must
  treat it as a hard prerequisite for `mv_player_profiles_v2`.

4.2 Player profile matching order (deterministic)
Priority 1: `player_crossref.tm_player_id` exact match.

Priority 2: canonical team-constrained name match:
- Resolve roster team via `team_canonical`:
  - `squad_rosters.team_id` -> `team_canonical.fbref_team_id` -> `tm_team_id` / `tm_name`.
- Candidate condition:
  - same `league_id`
  - player normalized name exact
  - and (`market_values.team_id = team_canonical.tm_team_id`
         or normalized TM team name = canonical TM alias)

Priority 3: name-only fallback (guarded):
- only when unique candidate in same league
- if multiple candidates: do not pick, keep NULL (avoid false positives)
- Known limitation v1:
  - Mid-season transfer lag between `squad_rosters` (FBref) and `market_values`
    (Transfermarkt) can create temporary team inconsistency for `name_only_unique`
    matches (profile team from roster, value/image from updated TM row).
  - Mitigation: flag all `matched_by = 'name_only_unique'` rows for manual review
    after each transfer window.

Tie-breaker order (when >1 candidate in same priority):
1) exact `tm_team_id` hit
2) highest `market_value_numeric`
3) most recent `loaded_at`

4.3 Team profile matching improvement
- For `mv_team_profiles`, prefer canonical mapping:
  - `standings.team_id` -> `team_canonical.fbref_team_id` -> `tm_team_id`
  - join `team_metadata.team_id = tm_team_id` first
- Keep normalized team name join only as fallback.

4.4 Add match quality columns in MV outputs
For player profile MV v2:
- `matched_by` values:
  - `crossref_tm_id`
  - `canonical_team_name_exact`
  - `name_only_unique`
  - `unmatched`
- `match_confidence` values:
  - 1.00 for exact ID
  - 0.90 for canonical team-constrained name
  - 0.60 for unique name-only
  - 0.00 for unmatched (convention: never use NULL)

5) Implementation Plan
Phase A: Diagnostics
- Build analysis query for ambiguous names per league.
- Baseline current coverage and known mismatch sample.
- Mandatory baseline SQL before Phase B:
  ```sql
  SELECT
      COUNT(*) FILTER (WHERE player_image_url IS NOT NULL) * 100.0 / COUNT(*) AS image_coverage_pct,
      COUNT(*) FILTER (WHERE market_value_numeric IS NOT NULL) * 100.0 / COUNT(*) AS value_coverage_pct
  FROM mv_player_profiles;
  ```
- Record baseline outputs in rollout notes; Phase C non-regression checks must compare
  against these exact baseline values.

Phase B: Build v2 MVs (shadow)
- Create `mv_player_profiles_v2` and `mv_team_profiles_v2` with new matching logic.
- Add indexes equivalent to current MVs.

Phase C: Diff validation
- Compare old vs v2 on:
  - coverage rate (`player_image_url` not null)
  - market value fill rate
  - suspicious changes list using fixed thresholds:
    - `market_value_numeric` change > 50% vs old MV
    - matched team differs from `squad_rosters.team_id`
    - `matched_by` downgrade from `crossref_tm_id` to lower-priority strategy
- Manual review sample (at least 50 changed rows).

Phase D: Adoption
- Switch downstream to v2 via feature flag.
- Keep fallback to old MVs for one release window.

Phase E: Cleanup
- Rename/swap v2 as default and retire old definition.

8) Risks & Mitigations
Risk A: Over-strict matching lowers coverage.
- Mitigation: keep guarded name-only unique fallback.

Risk B: Canonical table stale causes misses.
- Mitigation: refresh `team_canonical` before MV refresh in pipeline order.

Risk C: More complex MV increases refresh time.
- Mitigation: precompute candidate helper MV and add proper indexes.
